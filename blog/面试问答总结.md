### Q3: 为什么选 DECIMER 模型做化学结构式识别？对比 OSRA 等其他模型的优势是什么？

3. 场景覆盖：包含复杂结构、多环化合物、离子化合物等
4. 输出测试报告，作为技术选型决策依据

### Q4: 90% 以上识别准确率的测试集规模、数据来源、评估指标是什么？

**A**: 项目中没有任何测试数据或评估记录。仅有 3 张测试图片用于基本功能验证。

"90% 识别准确率"引用自 DECIMER 原论文，非项目实测。

**改进方案**：建立评估体系
1. **测试集构建**：
   - 从高中化学教材中选取 200 个常见分子结构
   - 包含不同质量（清晰/模糊）、不同来源（手写/打印/截图）
   - 标注正确的 SMILES 作为 ground truth

2. **评估指标**：
   - 完全匹配率（Exact Match）：SMILES 完全一致
   - Levenshtein 距离：编辑距离相似度
   - 分子级别准确率：识别出的分子结构是否正确（RDKit 验证）

3. **自动化测试脚本**：
   ```python
   # 批量测试 + 生成评估报告
   def evaluate_decimer(test_images, ground_truth):
       results = []
       for img, gt_smiles in test_images:
           pred_smiles = predict_SMILES(img)
           results.append({
               'exact_match': pred_smiles == gt_smiles,
               'valid': Chem.MolFromSmiles(pred_smiles) is not None,
               'levenshtein': distance(pred_smiles, gt_smiles)
           })
       return generate_report(results)
   ```

### Q5: RDKit ETKDGv3 + MMFF 力场优化有机分子的完整代码流程是什么？

**A**: 完整流程如下：

1. **判断分子类型**：先判断是否为无机分子（基于元素组成和结构模式）
2. **ETKDGv3 生成初始构型**：使用 ETKDGv3 算法，随机种子设为 42 保证可复现
3. **MMFF 力场优化**：优先尝试 MMFF94（适合有机分子），失败则回退 UFF
4. **输出格式**：转换为 PDB/SDF 格式返回前端

**关键参数**：使用实验扭转角偏好、应用化学知识、最大迭代 500 次

### Q6: ASE 封装自动化计算工作流，解决了什么研发痛点？全流程自动化步骤是什么？

**A**: 解决的核心痛点：无机分子（如 SF₄、PF₅、XeF₂ 等）的 3D 结构无法用传统分子力学准确生成。

**全流程自动化步骤**：
1. 原子对象构建：从 RDKit Mol 提取原子符号，创建 ASE Atoms 对象
2. 初始坐标获取：优先使用 RDKit ETKDG 生成的坐标作为初值，失败则用线性排列
3. 设置计算器：GFN2-xTB 半经验方法（速度快，精度适中）
4. 几何优化：BFGS 优化器，收敛标准为力 < 0.05 Hartree/Bohr，最大步数 1000
5. 坐标回写：将优化后的坐标写回 RDKit Mol 对象，输出标准 PDB/SDF 格式

## AI-Assistant 语音助手

### Q8: 为什么选 Electron？IPC 通信遇到什么问题？

**A**: 选择 Electron 的原因：

1. **跨平台需求**：一次开发，同时支持 Windows/macOS/Linux
2. **技术栈复用**：前端使用熟悉的 Web 技术
3. **Web Audio API**：原生支持音频采集
4. **Python 集成**：Node.js 可通过子进程调用 Python 脚本

**IPC 通信**：项目未使用 Electron IPC，而是采用 **HTTP + subprocess** 架构：

```
Electron 前端 → HTTP 请求 → Python Server (8765端口) → FunASR/GLM
```

**原因**：简化 Python 集成、方便独立调试、避免 IPC 传递大音频数据的性能问题

### Q9: 子进程调 Whisper 的原因？95% 准确率的测试环境是什么？

**A**: AI-assistant 项目有两个版本：旧版使用 Whisper（子进程调用），新版使用 FunASR（HTTP Server）。

**子进程调用 Whisper 的原因**：
- Whisper 是 Python 库，无法在 Node.js/Electron 中直接运行
- 必须通过 Python 解释器执行
- 进程隔离带来更好的错误处理和内存管理

**95% 准确率**：这是 OpenAI Whisper 官方论文声称的基准测试结果，非项目实测。

**改进方案**：建立语音识别评估体系
1. **测试数据**：收集 100 条中文语音样本（不同口音、语速、噪音环境）
2. **评估指标**：字错率 CER、句错率 SER
3. **对比测试**：Whisper vs FunASR 在相同数据上的表现
4. **生成报告**：记录不同场景下的准确率差异，为模型选择提供依据

### Q13: RAG 检索效果不理想，如何引入向量数据库提升？

**A**: 当前关键词匹配方案的局限性：无法理解语义相似性，召回率有限。

**改进方案**：集成 ChromaDB 向量数据库

**步骤**：
1. **向量化模型选择**：使用 sentence-transformers 的中文模型（如 `paraphrase-multilingual-MiniLM-L12-v2`）

2. **离线索引构建**：
```python
from sentence_transformers import SentenceTransformer
import chromadb

# 初始化
embedder = SentenceTransformer('模型路径')
client = chromadb.PersistentClient(path="./chroma_db")
collection = client.get_or_create_collection("textbook")

# 批量索引
for chunk in textbook_chunks:
    embedding = embedder.encode(chunk['content'])
    collection.add(
        ids=[chunk['id']],
        embeddings=[embedding.tolist()],
        documents=[chunk['content']],
        metadatas=[chunk['metadata']]
    )
```

3. **在线检索**：
```python
# 查询
query_embedding = embedder.encode(question)
results = collection.query(
    query_embeddings=[query_embedding.tolist()],
    n_results=5
)
```

4. **混合检索**：向量检索 + 关键词检索融合，提升准确率
